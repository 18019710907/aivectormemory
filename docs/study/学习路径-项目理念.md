# 学习路径：AIVectorMemory 项目理念

> 个人学习笔记。当你让 AI「研究」某类东西时，通常需要产出文档——本文即「项目理念」的研究产出。

---

## 为什么要写这份文档

理解一个项目的**理念**（为什么做、希望怎样被用），比单看代码更容易抓住设计意图。本文档给出**从哪入手、按什么顺序读**，便于后续自己或他人快速掌握 AIVectorMemory 的哲学与用法。

---

## 第一步：搞清楚「为什么要做」—— README

先读 **根目录 `README.md`**（或中文版 `docs/README.zh-CN.md`），重点三块：

### 1. 开篇两段（约 22–24 行）

- **传统做法的问题**：CLAUDE.md / MEMORY.md 文件式记忆的缺陷——文件越写越长、每次全量注入烧 token、只能关键词匹配、多项目共用一个文件会串味、没有任务追踪、截断与维护成本等。
- **本项目的思路**：向量库 + 语义检索、按需召回、多项目隔离、内置 issue/task、全本地。

### 2. Core Features 表格

每条都在表达理念的一个侧面：跨会话记忆、语义搜索、省 token、任务驱动、本地、多 IDE、多项目隔离、去重合并等。通读一遍即可建立「产品主张」的整体印象。

### 3. Architecture 图（约 64–90 行）

- 看清：IDE ↔ MCP ↔ 工具层（remember/recall/task/status…）↔ 嵌入引擎 ↔ SQLite+向量。
- 理解：记忆与任务都在服务侧，按需调用，不靠大段粘贴上下文。

**第一步收尾**：能回答——这个项目在解决什么问题？和「文件式记忆」的根本区别是什么？

## 第二步：搞清楚「希望 AI 怎么用」—— 工作规则（CLAUDE.md）

理念的**落地**在 **`CLAUDE.md`**（与 `.cursor/rules/aivectormemory.md` 内容一致）。建议按节精读：

### §1 新会话启动

先 recall 项目知识、再 recall 用户偏好、再读 status；有阻塞就先汇报再等。  
→ 体现：**记忆是按需加载的，不是整份文件塞进上下文**。

### §2 收到消息后的处理流程（A→I）

分类（闲聊 / 纠正 / 偏好 / 问题）→ 问题必先 `track create` → 排查（其间用 recall 查踩坑）→ 给出方案并**设阻塞等确认** → 用户确认后再改代码 → 验证 → 归档 + 有价值就 remember。  
→ 体现：**问题必记录、先方案后动手、关键节点必须用户确认、修完要沉淀成记忆**。

### §3 阻塞规则

何时设阻塞、何时才能清阻塞、禁止自行清、不猜用户意图。  
→ 体现：**AI 不替用户做决定，会话切换/续接后仍以阻塞为准**。

### §4 track / §6 task

- **track**：问题从发现到归档的完整链路，字段要填满。
- **task**：多步骤需求走 spec（requirements → design → tasks）再拆任务执行。

→ 体现：**开发过程被结构化、可追溯、可跨会话延续**。

### §7 记忆质量

tags 规范、踩坑要写现象+根因+正确做法等。  
→ 体现：**记忆是给「以后 recall」用的，要可检索、可复用**。

**第二步收尾**：能回答——产品理念不只是「有个向量记忆」，而是「记忆 + 追踪 + 状态 + 任务」一整套协作方式。

## 第三步：工具设计（可选）

在 **README 的「8 MCP Tools」** 里扫一遍：`remember` / `recall` / `forget` / `status` / `track` / `task` / `auto_save` / `readme` 等各自负责什么、参数大致什么意思。  
不必背，建立「记忆、状态、追踪、任务」各由哪些工具支撑即可。

## 第四步：动手或看代码（可选）

- **动手**：在某个小项目里 `run install` 装一次，按 README 用几回 remember/recall/track，感受「先记录再修」「先方案再执行」的流程。
- **看代码**：从 `aivectormemory/server.py` 看工具注册，再挑一个工具（如 `tools/remember.py` + `tools/recall.py`）看一次「写入 → 向量化 → 存储」和「查询 → 向量检索 → 返回」的路径，对「语义检索」会有更具体的理解。
